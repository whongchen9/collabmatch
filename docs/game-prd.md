# 需求匹配 · 游戏化外壳 PRD

> 版本 v0.1 · 作者：许清楚 · 目标：给现有 Web 应用加一层"驿站"2D 游戏场景，看似网游实则联通现实协作

---

## 1. 产品目标

在现有"需求匹配"应用之上，加一层像素风 2D 游戏化外壳（Canvas 渲染），将表单/列表 UI 转化为一个可走动、可交互的古代驿站场景，降低使用门槛、提升留存，但**所有底层协作逻辑不变**。

---

## 2. 核心用户故事

| # | 故事 | 验收标准 |
|---|------|----------|
| US1 | 作为新用户，我进入驿站场景，看到一个像素角色（我），可以用键盘走动探索，感觉像在玩游戏而不是填表 | 角色可用 WASD/方向键移动，场景有边界碰撞，有简单动画帧 |
| US2 | 作为有需求的用户，我走到 NPC 小二哥前按 E 对话 → 弹出 AI 对话面板（就是现在的 CollabAI），描述需求、生成方略 | 对话完返回游戏场景，已生成的需求可继续操作 |
| US3 | 作为找项目的 Builder，我走到告示墙前 → 弹出需求广场列表（调用现有 API），可查看详情、申请参与 | 告示墙数据与现有广场同步，申请流程复用现有逻辑 |
| US4 | 作为被推荐匹配的用户，我看到一个侠客 NPC 出现在驿站内，走近交互 → 展示匹配卡片，点击邀请或跳过 | 侠客出现/消失由匹配 API 驱动，交互后触发邀请流程 |
| US5 | 匹配成功后，我和对方角色在酒桌前"入座共饮"动画展示，然后跳转协作群组 | 动画 2-3 秒，之后自动进群组页 |

## 3. 功能需求

### P0 — 必须做（最小可玩）

| ID | 功能 | 说明 |
|----|------|------|
| F1 | Canvas 游戏引擎 | 单 Canvas，像素风 32x32 tile，自研轻量渲染循环（requestAnimationFrame），不做 Phaser 等重框架 |
| F2 | 驿站场景 | 1 个场景（室内客栈），约 20x15 tile，含背景层 + 物件层 + 角色层 |
| F3 | 玩家角色 | 像素小人（16x24 或 32x32），4 方向精灵帧（站/走各 3-4 帧），WASD/方向键移动，碰撞检测 |
| F4 | NPC 小二哥 | 固定位置，走近按 E → 拉起 CollabAI 对话面板（overlay 在 Canvas 上方），对话逻辑完全复用现有 |
| F5 | 告示墙（需求广场） | 固定位置，走近按 E → 弹出需求广场浮层，数据调现有 `GET /api/requirements`，交互复用现有卡片组件 |
| F6 | 场景/UI 切换 | 场景常驻背景，弹窗/面板以 DOM overlay 形式浮在 Canvas 上，关闭后回到游戏场景 |

### P1 — 应该做（增强体验）

| ID | 功能 | 说明 |
|----|------|------|
| F7 | 匹配侠客 NPC | 定时/事件触发在场景中随机位置生成"侠客"角色（像素小人 + 名字气泡），代表匹配推荐的人；走近交互 → 弹窗展示匹配详情，可邀请 |
| F8 | 匹配成功动画 | 双方角色传送到酒桌区域，播放"入座→斟酒→共饮"动画序列（约 2-3 秒），然后自动打开群组 |
| F9 | 个人名片入口 | 场景角落放"铜镜"物件，点击 → 弹出个人名片面板（复用现有 `renderProfile`） |
| F10 | 音效 | 走动脚步声、交互提示音、匹配成功音效（用 Web Audio API，简单合成，不依赖音频文件） |

### P2 — 锦上添花

| ID | 功能 | 说明 |
|----|------|------|
| F11 | 多个场景 | 驿站 → 野外 → 集市等，场景切换动画 |
| F12 | 角色换装 | 根据角色/领域显示不同外观（技术=蓝袍、设计=青衫等） |
| F13 | 知识库书架 | 场景内书架物件，点击进入"来时路"知识库 |
| F14 | 多人同屏 | 同场景内能看到其他在线用户的角色走动（需要 WebSocket） |

## 4. 场景与 NPC 设计

### 4.1 驿站场景布局（约 20x15 tile）

```
┌──────────────────────────────────────┐
│  [窗]       [告示墙]        [窗]     │  ← 北墙
│                                      │
│  酒桌1          酒桌2      [书架]    │  ← P1/P2 物件
│  ╔══╗          ╔══╗                 │
│  ║  ║          ║  ║                 │
│                                      │
│    [小二哥 NPC]          [铜镜]      │  ← NPC + 名片入口
│     👤                           🪞   │
│                                      │
│          入口（玩家出生点）           │  ← 南面入口
└──────────────────────────────────────┘
```

### 4.2 NPC/交互点定义

| 名称 | 位置 | 外观 | 交互方式 | 映射功能 |
|------|------|------|----------|----------|
| 小二哥 | 驿站中央偏左 | 像素店小二形象，待机有呼吸/踱步动画 | 走近 2 tile 内按 E | CollabAI 对话 |
| 告示墙 | 北墙中央 | 木牌告示板，贴有几张需求纸片（静态） | 走近按 E | 需求广场列表 |
| 侠客 NPC | 随机/固定刷新 | 像素角色 + 头顶悬浮名字，偶尔闲逛 | 走近按 E | 匹配推荐卡片 |
| 铜镜 | 东南角 | 铜镜台 | 走近按 E | 个人名片 |
| 酒桌 | 中央区域 2 张 | 方桌 + 长凳 | 无法交互（动画触发点） | 匹配成功动画位 |
| 书架 | 东北角（P2） | 书架 | 走近按 E | 来时路知识库 |

### 4.3 视觉风格

- **画风**：像素风（pixel art），参考 16-bit RPG（如早期最终幻想/轩辕剑）
- **色板**：暖色调（木色、纸色、灯笼红），营造温馨客栈氛围
- **tile 大小**：32x32px，场景分辨率约 640x480 基础，自适应缩放
- **角色大小**：玩家/NPC 约 24x32px（在 32x32 tile 内）

## 5. 与现有系统的集成点

### 5.1 技术集成

| 现有模块 | 游戏化接入方式 | 改动量 |
|----------|---------------|--------|
| `index.html`（单文件 SPA） | 新增 `<canvas id="game-canvas">` + 游戏 JS 模块，与现有 DOM 共存；在 `#app` 根节点内切换显示模式 | 中 |
| CollabAI 对话 | 小二哥交互→调用 `switchSection('home')` 或直接打开对话面板 overlay；后端 API 零改动 | 低 |
| 需求广场 `renderSquare()` | 告示墙交互→弹出需求列表 Modal，复用现有 API 和渲染逻辑 | 低 |
| 匹配 `matchForward()` | 匹配结果驱动侠客 NPC 生成；API 零改动，仅前端消费现有数据 | 低 |
| 邀请/建群 `sendInvite()` | 侠客交互弹窗→调用现有邀请逻辑；匹配成功动画→调用 `switchSection('groups')` | 低 |
| 个人名片 `renderProfile()` | 铜镜交互→弹出名片面板 | 低 |
| `api-bridge.js` | 零改动，全部复用 | 无 |

### 5.2 数据流示意

```
用户按键输入 → Canvas 渲染循环（游戏引擎）
                    ↓ (按 E 靠近交互点)
            触发 DOM overlay 弹窗
                    ↓
        调用现有 JS 函数 (switchSection / openReqDetail / applyReq ...)
                    ↓
              api-bridge.js → Server API
                    ↓
              返回数据 → 更新 DOM overlay → 关闭回到 Canvas
```

### 5.3 文件改动

| 文件 | 改动 |
|------|------|
| `index.html` | 新增 Canvas 元素 + 游戏 overlay CSS + 游戏 JS 模块（inline 或外部脚本） |
| `scripts/game/`（新建） | 游戏引擎核心：`engine.js`（渲染循环/输入）、`scene.js`（场景数据）、`player.js`（角色控制）、`npc.js`（NPC 逻辑）、`sprites.js`（精灵动画） |
| 后端 | **零改动** |

## 6. 待确认问题

| # | 问题 | 选项/建议 |
|---|------|----------|
| Q1 | 游戏模式是默认入口，还是和现有 UI 可切换？ | **建议默认入口**，右上角保留"退出游戏模式"按钮回到传统 UI |
| Q2 | 精灵/像素素材从哪来？ | 建议用程序化绘制（Canvas 直接画像素小人，免素材版权）+ 可后续替换为美术素材 |
| Q3 | 移动端如何操作？ | 虚拟摇杆 + 点击移动，但**P0 先做桌面端**（键盘操作），移动端 P1 适配 |
| Q4 | 场景内是否显示其他在线用户？ | P0 不做（仅本地单人），P2 可加 WebSocket 多人同屏 |
| Q5 | 侠客 NPC 刷新频率？ | 建议每次进入场景时检查最新匹配结果，有推荐则生成 1-2 个侠客，停留 30 秒后消失 |
| Q6 | 游戏画布覆盖整个页面还是嵌入式窗口？ | 建议全屏 Canvas 背景 + DOM overlay 弹窗，传统 UI 隐藏但保留逻辑 |
| Q7 | 是否需要登录后才能进入场景？ | 是，进入游戏场景前先走现有登录流程，未登录显示登录页 |
| Q8 | 像素动画帧用雪碧图还是程序绘制？ | 建议程序绘制（Canvas drawImage + 切换色块），避免外部资源加载，首屏更快 |

## 7. 技术风险

| 风险 | 影响 | 缓解 |
|------|------|------|
| 游戏循环与 DOM 渲染冲突 | 性能问题、卡顿 | Canvas 和 DOM overlay 分层，游戏循环暂停在弹窗打开时 |
| 单文件 HTML 过大 | 维护困难 | 游戏模块拆到 `scripts/game/` 目录，用 `<script>` 标签加载 |
| 移动端体验差 | 触屏交互不合适 | P0 不做移动端，桌面端优先 |
| 像素美术素材为零 | 视觉差 | 全部程序化绘制（矢量 → 像素），后续可替换 |
